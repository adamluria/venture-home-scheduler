## Architectural decisions
- **Why a separate `?name=` SOSL branch instead of LIKE on Name?** SOSL hits Salesforce's indexed search, handles tokens (first+last), and natively spans Lead and Contact in one call. SOQL `Name LIKE '%term%'` would require two queries and couldn't use the index. Tradeoff: SOSL has indexing lag — very-new Leads (<5 min old) may not surface. Reps can still find them by phone/email exact match.
- **Why prefer MobilePhone?** Day-of contact for sales appointments is overwhelmingly via mobile/SMS. Office `Phone` is the fallback.
- **Why not auto-fetch on `?lead=LEADID` alone?** The existing button always passes all fields in the URL. Adding a backend round-trip would be defensive code with no current user. If we ever want to support stale-URL refresh, the `/api/sfdc/lead/:id` endpoint is ready — wire it into the `sfdcDefaults` effect in `app.jsx`.

## Risks / known limitations
- **SOSL indexing lag**: brand-new Leads (created in the last few minutes) may not appear in name search. Phone/email exact-match is the workaround.
- **`?leadId=` deep-link variant not implemented**: only the existing `?lead=...&name=...&phone=...&...` URL pattern works. ID-only fallback is a future improvement.
- **Lead Source whitelist**: if SF returns a `LeadSource` value not in `leadSources.js`, the picker silently keeps the form's default (`'paid'`). Worth a one-time audit against the 495-value picklist.

## [Tier 1] Customer interaction history

### `GET /api/sfdc/customer-history?phone=XXX` (server.js, ~line 1104)
Aggregates all SF interactions matched by phone (last-10-digits via SOSL `IN PHONE FIELDS`):
- **Step 1:** SOSL phone search → finds Leads + Contacts (limit 25 each)
- **Step 2-6 (parallel):** Opportunities (by AccountId), Tasks (WhoId/WhatId), Events (WhoId/WhatId), ContentDocumentLinks
- **Step 7:** ContentNote details for any linked docs + legacy Note fallback

Each sub-query is wrapped in try/catch and returns [] on permission errors so a missing ContentNote read permission doesn't kill the whole endpoint. Errors collected into a top-level `errors` array for debugging.

**Response shape:**
```
{
  matchedDigits: "5551234567",
  summary: { leadCount, contactCount, oppCount, callCount, taskCount, eventCount, noteCount, firstContact, lastContact },
  leads: [{ id, name, status, source, isConverted, description, ownerName, ... }],
  contacts: [...], opportunities: [...],
  tasks: [{ id, subject, type, isCall, callDurationSeconds, callDisposition, description, ... }],
  events: [...], notes: [{ id, title, preview, kind: 'content_note'|'legacy_note', ... }],
  errors?: [{ label, status, snippet }]
}
```

### Architectural notes
- **Why phone-only matching?** User chose this — phone is the most stable identity across name changes (married, typos), and SOSL's `IN PHONE FIELDS` is index-backed and handles formatting variants natively. Email matching could be added later by extending the SOSL.
- **Why limit Tasks to 50, ContentNotes to 30, Events to 25?** Bounds the response size for the picker UI. Older records show as "+N older" hints; click-through to SF for full history.
- **Transcripts:** SF Task.Description is the surfaced field. Real "transcripts" (multi-paragraph dialogue) typically live there for orgs with Aircall/Service Cloud Voice. AI summarization is a future enhancement — code is shaped to add a `summary` field to call records when that's wired.
- **No new endpoints called from app boot.** History only fetches on user-triggered chevron expand — zero impact on initial load.

## [Tier 1] Per-rep Salesforce auth + token persistence

### `sfdcAuth.js` (new, root-level Node module)
Replaces the previous `app.locals.sfdc` single-shared-token pattern.

- **Storage**: in-memory `Map` keyed by session id. Designed as a swap point for Firestore in production multi-instance Cloud Run.
- **Cookie**: `vhs_sfdc_session`, HTTP-only, SameSite=Lax, Secure in prod, 7-day TTL.
- **Refresh-on-401**: `sfdcFetch(req, url, options)` wraps SF REST calls; if SF returns 401, it uses the session's `refresh_token` to get a new access token, updates the session, and retries once. If refresh fails, the session is purged and the next call returns 401 to the frontend (which redirects to /auth/salesforce).
- **Identity**: on auth callback, fetches the SF identity URL (`tokens.id`) to capture `sfEmail`, `sfUserId`, `sfDisplayName` for the whoami endpoint.

### `server.js` changes
- **Middleware**: `app.use(loadSfdcSession)` runs on every request, attaches `req.sfdc` if the cookie maps to a valid session.
- **OAuth callback**: now creates a per-rep session, sets the HTTP-only cookie, and honors a `?return=<path>` query param so deep links survive the redirect.
- **All 13 endpoint auth-checks** rewritten from `const sfdc = app.locals.sfdc; if (!sfdc?.accessToken) ...` to `if (!req.sfdc) ...; const sfdc = req.sfdc;`.
- **New endpoints**: `GET /api/sfdc/whoami` (frontend uses this for auth state UI), `POST /api/sfdc/logout` (clears session + cookie).
- **customer-history endpoint** now uses `sfdcFetch` for all sub-queries — the most fetch-heavy endpoint gets refresh-on-401.
- **Backward-compat**: `app.locals.sfdc` is still seeded on auth so any not-yet-refactored future code paths keep working during the rollout. Safe to remove later.

### `src/components/SfdcAuthBanner.jsx` (new)
Top-of-dashboard auth status indicator. Polls `/api/sfdc/whoami` on mount and on window-focus. When unauthed: shows a non-blocking accent banner with "Connect Salesforce" CTA → redirects to `/auth/salesforce?return=<here>`. When authed: collapses to a small green pill showing the rep's email + a sign-out icon.

### Architectural notes
- **Why per-rep, not service account?** Adam chose this. Each rep authenticates separately via SF OAuth; their tokens stay isolated in the session store. This means each rep's queries run as themselves in SF (correct field-level security, audit trail, etc).
- **Why in-memory now?** Sandbox testing only needs a single Cloud Run instance. Code is shaped so the swap to Firestore is touching only the `sessions` Map declaration in `sfdcAuth.js`.
- **Why not wrap every existing fetch with `sfdcFetch`?** Tradeoff: wrapping all 16 fetch calls is high-churn, low-value. Instead, the customer-history endpoint (the one most likely to hit a stale token because it fires 6+ parallel queries) is wrapped, and any 401 from other endpoints surfaces to the frontend, where the auth banner lets the rep re-auth in one click. Future endpoints should use `sfdcFetch` from the start.

### Production cutover constraints
- **`--max-instances 1`** required until Firestore swap. Documented in `docs/salesforce-sandbox-setup.md` §2.3.
- **Token refresh works**, but the SF refresh_token is in process memory — a Cloud Run revision change (deploy) wipes all sessions; reps re-auth on next visit.
- **Single connected app** can't serve both sandbox and prod simultaneously — needs separate `SF_CLIENT_ID`/`SF_CLIENT_SECRET`. Sandbox test plan documented.
