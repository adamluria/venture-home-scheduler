## Files

### [Tier 1] `src/data/leadMapper.js` (new)
Single source of truth for SF Lead/Contact → form values. Accepts both raw SFDC REST shape (`FirstName`, `LastName`, `MobilePhone`, `Street`, `PostalCode`) AND the normalized search-endpoint shape (`firstName`, `mobilePhone`, `address`, `zip`).

**Mapping rules**:
- `customer` ← `FirstName + ' ' + LastName`, fall back to `Name`
- `phone` ← `MobilePhone || Phone` (mobile preferred for day-of contact)
- `email` ← `Email`
- `address` ← `[Street, City, State].filter(Boolean).join(', ')` — keeps `parseAddressPart()` working for SlotSuggestions
- `zipCode` ← `PostalCode` (or `MailingPostalCode` for Contacts)
- `leadSource` ← only carried if value matches `LEAD_SOURCES` whitelist
- `sfdcLeadId` ← carried only for Lead records (not Contacts) so `syncAppointmentToSFDC` auto-converts on save

## Architectural decisions
- **Why a separate `?name=` SOSL branch instead of LIKE on Name?** SOSL hits Salesforce's indexed search, handles tokens (first+last), and natively spans Lead and Contact in one call. SOQL `Name LIKE '%term%'` would require two queries and couldn't use the index. Tradeoff: SOSL has indexing lag — very-new Leads (<5 min old) may not surface. Reps can still find them by phone/email exact match.
- **Why prefer MobilePhone?** Day-of contact for sales appointments is overwhelmingly via mobile/SMS. Office `Phone` is the fallback.
- **Why not auto-fetch on `?lead=LEADID` alone?** The existing button always passes all fields in the URL. Adding a backend round-trip would be defensive code with no current user. If we ever want to support stale-URL refresh, the `/api/sfdc/lead/:id` endpoint is ready — wire it into the `sfdcDefaults` effect in `app.jsx`.

## Risks / known limitations
- **SOSL indexing lag**: brand-new Leads (created in the last few minutes) may not appear in name search. Phone/email exact-match is the workaround.
- **`?leadId=` deep-link variant not implemented**: only the existing `?lead=...&name=...&phone=...&...` URL pattern works. ID-only fallback is a future improvement.
- **Lead Source whitelist**: if SF returns a `LeadSource` value not in `leadSources.js`, the picker silently keeps the form's default (`'paid'`). Worth a one-time audit against the 495-value picklist.

## [Tier 1] Customer interaction history

### `GET /api/sfdc/customer-history?phone=XXX` (server.js, ~line 1104)
Aggregates all SF interactions matched by phone (last-10-digits via SOSL `IN PHONE FIELDS`):
- **Step 1:** SOSL phone search → finds Leads + Contacts (limit 25 each)
- **Step 2-6 (parallel):** Opportunities (by AccountId), Tasks (WhoId/WhatId), Events (WhoId/WhatId), ContentDocumentLinks
- **Step 7:** ContentNote details for any linked docs + legacy Note fallback

Each sub-query is wrapped in try/catch and returns [] on permission errors so a missing ContentNote read permission doesn't kill the whole endpoint. Errors collected into a top-level `errors` array for debugging.

**Response shape:**
```
{
  matchedDigits: "5551234567",
  summary: { leadCount, contactCount, oppCount, callCount, taskCount, eventCount, noteCount, firstContact, lastContact },
  leads: [{ id, name, status, source, isConverted, description, ownerName, ... }],
  contacts: [...], opportunities: [...],
  tasks: [{ id, subject, type, isCall, callDurationSeconds, callDisposition, description, ... }],
  events: [...], notes: [{ id, title, preview, kind: 'content_note'|'legacy_note', ... }],
  errors?: [{ label, status, snippet }]
}
```

### `src/components/LeadHistoryPanel.jsx` (new)
Renders the response inline. Sections (each collapsible): Summary chips → Prior Leads → Opportunities → Calls → Meetings → Notes → Other Activity. Calls section is the "transcripts" surface — Task.Description holds the call notes; CallDurationInSeconds renders as "X min". Pills color-code Lead.Status (won/lost/working) and Opp.StageName (won/lost/in-flight).

### `src/components/LeadPicker.jsx` (modified)
Added expand chevron per result. Click row body = select (existing behavior). Click chevron = toggle inline `LeadHistoryPanel` without selecting. Disabled if the record has no phone (chevron greyed out, tooltip "No phone on file"). Phone passed to history panel is the last-10-digits-normalized form.

### Architectural notes
- **Why phone-only matching?** User chose this — phone is the most stable identity across name changes (married, typos), and SOSL's `IN PHONE FIELDS` is index-backed and handles formatting variants natively. Email matching could be added later by extending the SOSL.
- **Why limit Tasks to 50, ContentNotes to 30, Events to 25?** Bounds the response size for the picker UI. Older records show as "+N older" hints; click-through to SF for full history.
- **Transcripts:** SF Task.Description is the surfaced field. Real "transcripts" (multi-paragraph dialogue) typically live there for orgs with Aircall/Service Cloud Voice. AI summarization is a future enhancement — code is shaped to add a `summary` field to call records when that's wired.
- **No new endpoints called from app boot.** History only fetches on user-triggered chevron expand — zero impact on initial load.
