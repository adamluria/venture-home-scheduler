## [Tier 1] Appointment__c Status__c picklist ‚Äî canonical values

Audit found the Status__c value space was inconsistent across three layers (frontend, server I/O, setup doc). Locked it down before the object is created in sandbox so we don't bake in drift.

### Canonical SF picklist (defined in `docs/salesforce-sandbox-setup.md` ¬ß1.3)

```
Scheduled, Confirmed, Completed, Closed Won, Canceled, No Show
```

- American spelling (`Canceled`)
- `No Show` with **space**, not hyphen ‚Äî matches the existing 5 SOQL queries that read it
- `Closed Won` added to the picklist; the SOQL reads at server.js:519/563/592/620/649 already filter on it (was previously a phantom value not in the doc)

### Frontend wire format ‚Üí SF mapping

Frontend keeps lowercase / kebab-case as its idiomatic `apt.status` representation. Server-side mapper translates on the wire. Defined as `STATUS_FE_TO_SF` in `server.js` (top of the Appointment__c CRUD block):

| frontend `apt.status` | SF `Status__c` |
|---|---|
| `scheduled` | `Scheduled` |
| `confirmed` | `Confirmed` |
| `completed` | `Completed` |
| `closed-won` | `Closed Won` |
| `canceled` | `Canceled` |
| `no-show` | `No Show` |

Applied in:
- POST `/api/sfdc/appointment` (line ~820) ‚Äî `toSfStatus(apt.status) || 'Scheduled'`
- PATCH `/api/sfdc/appointment/:id` (line ~865) ‚Äî wraps `updates.Status__c` if present
- DELETE `/api/sfdc/appointment/:id` (line ~909) ‚Äî already wrote `'Canceled'`, untouched

Frontend-only statuses (`needs-reschedule`, `rescheduled`, `disqualified`) are left unmapped ‚Äî they fall through `toSfStatus` unchanged. If they ever need to sync, add to the map; for now they're local-only UI state and SF writes for those will fail picklist validation, which is the intended signal.

### Why server-side mapping vs. migrating frontend to Title Case

UI code references `apt.status` in ~25 places across LeaderboardView, AnalyticsView, AppointmentCard, DraggableDayView, SwimlaneView, SearchBar, slotSuggestionEngine, leadScoring, mockData. Migrating all of them to `'Closed Won'` (with space) splatters SF picklist semantics across the UI layer and breaks the natural lowercase JS convention. One server-side mapper is the seam.

## [Tier 1] Adjacent bugs fixed in same commit

**1. `slotSuggestionEngine.js:331` ‚Äî silent filter bug**
Was: `a.status !== 'cancelled'` (British double-L). Frontend uses American `'canceled'` everywhere ‚Äî this filter never fired. Cancelled appointments were counted toward today's load, inflating workload penalties. Fixed to `'canceled'`.

**2. `LeaderboardView.jsx:72,140` and `AnalyticsView.jsx:32,37,52,69` ‚Äî closed === completed conflation**
Was: `closed = filter(status === 'completed')`. Treated every sit as a close. SF semantics distinguish `Completed` (sat) from `Closed Won` (sat + signed). Fixed to `closed = filter(status === 'closed-won')`. AnalyticsView's `sits` recomputed as `completed + closed-won` (a closed-won deal was sat first).

**Demo-data side effect:** `mockData.js` has zero `'closed-won'` records. Leaderboard "closed" column will read 0 for every rep until either (a) mock records gain a `closed-won` status, or (b) real SF data flows in. Honest representation; flagged here so it's not surprising.
